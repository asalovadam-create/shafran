# SHAFRAN

Одностраничный сайт: живой таймер → честная очередь номеров → каталог платков.
Бэкенд — Python (FastAPI), фронтенд — один HTML-файл (CSS+JS внутри), PWA.

## Структура (всего 8 файлов — минимум, как и просили)

```
app.py                  # весь бэкенд: API, WebSocket, база, выдача номеров
requirements.txt        # 3 зависимости
Procfile                # команда запуска для Render
static/index.html       # весь сайт для посетителей (HTML+CSS+JS в одном файле)
static/admin.html       # мини-панель для запуска таймера
static/manifest.json    # PWA-манифест
static/sw.js            # service worker (мгновенная загрузка)
static/logo.png         # ЗАГЛУШКА — замените на свой логотип
static/bg.jpg           # ЗАГЛУШКА — замените на свою фотографию
static/icons/*.png      # ЗАГЛУШКИ иконок PWA — замените на свои
```

## Как это работает (главное — про честность номеров)

1. Админ на `/admin` вводит ключ и запускает таймер на N секунд.
2. Через WebSocket команда `timer_start` летит всем подключённым мгновенно —
   у всех участников таймер стартует одновременно.
3. Когда обратный отсчёт у конкретного человека в браузере доходит до нуля,
   у него появляется кнопка «Получить номер».
4. Нажатие бьёт в `POST /api/claim`. Внутри сервера стоит `asyncio.Lock`:
   пока один запрос не дописал номер в базу, следующий не начнёт выполняться.
   Поэтому даже если 50 человек нажмут в одну миллисекунду — номера уйдут
   строго по очереди 1, 2, 3… без повторов. Это проверено тестом:
   50 параллельных запросов → 50 уникальных номеров, ни одного дубля.
5. Повторное нажатие уже получившим номер ничего не ломает — просто
   возвращает тот же номер.

**Важно при деплое:** сервис должен работать **одним воркером**
(`--workers 1`). Это уже прописано в `Procfile`. Один процесс = один `Lock`
на всех = гарантия без дублей. Если когда-нибудь понадобится несколько
процессов (например, при очень большой нагрузке) — нужно будет перенести
блокировку на уровень базы данных (Postgres `SELECT … FOR UPDATE`), но для
20–50 участников это не требуется.

## Деплой на Render

1. Залейте эту папку в GitHub-репозиторий.
2. На render.com: **New → Web Service**, подключите репозиторий.
3. Runtime: Python 3. Build command: `pip install -r requirements.txt`.
   Start command можно не указывать — Render подхватит `Procfile`.
4. В Environment добавьте переменную `SHAFRAN_ADMIN_KEY` — свой секретный
   ключ администратора (иначе будет использован ключ по умолчанию из
   кода — обязательно замените его на свой перед стартом продажи!).
5. Диск: SQLite-файл `shafran.db` создастся автоматически рядом с `app.py`.
   На бесплатном плане Render диск не персистентный между деплоями —
   для одного «забега» (разового розыгрыша номеров) это не проблема,
   но если хотите держать историю участников между релизами — подключите
   Render Disk (Persistent Disk, платно) или замените SQLite на Render
   Postgres (бесплатный тариф тоже есть).

Админка доступна по адресу `ваш-сайт.onrender.com/admin`.

## Как загрузить свои картинки — чтобы сайт грузился мгновенно

**Фон (`static/bg.jpg`):**
- Формат: **WebP** (лучше) или JPEG с качеством 75–82%. WebP при том же
  визуальном качестве весит на 25–35% меньше.
- Размер: не больше **1600×2000 px** для мобильного портретного фона —
  бо́льшее разрешение браузер всё равно сожмёт под экран, а трафик
  потратится зря.
- Вес: старайтесь уложиться в **150–250 КБ**. Прогоните файл через
  squoosh.app (бесплатно, в браузере) — выберите WebP, качество ~80.
- Замените файл `static/bg.jpg` на свой (можно оставить .jpg или
  переименовать на .webp — тогда поправьте путь в `index.html` и `sw.js`
  в одном месте: `url('/static/bg.jpg')`).

**Логотип (`static/logo.png`):**
- Да, PNG с прозрачным фоном — то, что нужно. Если у вас логотип на белом
  фоне, уберите фон через remove.bg или в Figma/Photoshop (Magic Wand +
  Delete), экспортируйте в PNG-24 с альфа-каналом.
- Высота логотипа в шапке — 34px на экране, но экспортируйте файл заметно
  крупнее (например, высотой 300–400 px) — на Retina-экранах он должен
  оставаться чётким. Вес файла обычно получается 20–60 КБ — это нормально.

**Иконки PWA (`static/icons/`):** нужны 3 файла:
- `icon-192.png` — 192×192
- `icon-512.png` — 512×512
- `icon-maskable-512.png` — 512×512, но с логотипом по центру и запасом
  20% пустого поля со всех сторон (Android обрезает края в кружок/квадрат
  с закруглением — это и называется maskable).

## Почему сайт грузится «за миллисекунду»

- Весь фронтенд — **один HTML-файл**: браузеру не нужно делать отдельные
  запросы за CSS и JS, это самый быстрый вариант для одностраничника.
- `<link rel="preload" as="image" href="/static/bg.jpg">` — браузер начинает
  качать фон ещё до того, как дочитает остальной HTML.
- Шрифт Fraunces подключён с `media="print" onload="this.media='all'"` —
  это классический трюк «не блокировать отрисовку страницы шрифтом»:
  сайт сразу показывается системным шрифтом, а красивый шрифт подменяется
  бесшовно, когда догрузится.
- Service worker кэширует HTML/фон/логотип после первого визита — второй
  и все последующие визиты открываются из кэша устройства, без сети.
- Товарные фотографии подключены с `loading="lazy"` — грузятся только
  когда пользователь долистал до них, а не все разом при открытии сайта.

## Cloudinary и Россия — важный момент

Вы упомянули, что живёте в России и хотите, чтобы картинки «перековались»
(отдавались) быстрее. Честно предупреждаю:

- Cloudinary — сеть с глобальными CDN-нодами, но у части российских
  провайдеров случаются просадки скорости/задержки до зарубежных CDN
  (не заблокирован официально, но маршрутизация до ближайшей ноды не
  всегда идеальна, особенно у мобильных операторов).
- Обязательно используйте параметры трансформации **`f_auto,q_auto`**
  в URL картинки (я это уже сделал в примерах товаров) — Cloudinary сам
  отдаст WebP/AVIF при поддержке браузера и сожмёт без потери видимого
  качества. Это даёт основной прирост скорости.
- Добавляйте `w_600` (или нужную ширину) в URL — не отдавайте
  оригинал 4000px там, где на экране картинка шириной 300px.
- Если после запуска увидите, что для российских пользователей
  Cloudinary всё же медленный — рабочая альтернатива: объектное хранилище
  Selectel S3 (у вас уже есть аккаунт Selectel) с их же CDN — трафик
  внутри российской сети будет быстрее, чем до зарубежного CDN. Структуру
  кода менять не придётся — просто замените ссылки в массиве `PRODUCTS`
  внутри `static/index.html` на новые URL.

## Как редактировать каталог платков

Откройте `static/index.html`, найдите массив `PRODUCTS` в конце файла —
там простой список объектов `{ name, desc, img, inStock }`. Добавляйте,
удаляйте, меняйте местами — карточки перестроятся автоматически.
Специально нет корзины и цен — по ТЗ сайт информационный, «для вида».

## Проверено тестами перед сдачей

- ✅ 50 одновременных нажатий «Получить номер» → 50 уникальных номеров
  1…50, без единого дубля.
- ✅ Повторное нажатие уже получившим номер — не создаёт побочных эффектов.
- ✅ Попытка получить номер до открытия очереди — корректно отклоняется.
- ✅ Неверный ключ администратора — доступ запрещён.
- ✅ Сброс (`/api/admin/reset`) корректно обнуляет таймер и очередь для
  нового «забега».
- ✅ Проверены логи сервера на трейсбеки — чисто.

## Локальный запуск (для проверки перед деплоем)

```bash
pip install -r requirements.txt
uvicorn app:app --reload
# сайт:  http://127.0.0.1:8000
# админка: http://127.0.0.1:8000/admin  (ключ по умолчанию: shafran-admin-2026)
```

Перед продакшеном обязательно смените `SHAFRAN_ADMIN_KEY` через переменную
окружения — не оставляйте ключ по умолчанию.
